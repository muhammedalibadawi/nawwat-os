# Demo Flow (Internal)

## البيانات المطلوبة مسبقًا (minimum)
- tenant واحد نشط + فروع للمطاعم والصيدلية.
- Restaurants:
  - على الأقل: `طاولات` + `قائمة` + أصناف (بعضها بدون modifiers وبعضها ب modifiers) + seed كافٍ لإظهار تذاكر في KDS.
- Pharmacy:
  - batches متاحة + expiry dates (حتى يظهر FEFO/near/healthy بوضوح).
  - على الأقل patient/customer + seed يسمح بظهور queue في `Prescription` إن رغبت بتمثيل صرف الوصفة.
- WorkOS:
  - على الأقل Team Space + Project + Doc + Channel + Thread + بعض Notifications/Activities لتعبئة الشاشات.

## سيناريو مطاعم (POS → KDS → Payment)
1. افتح `/restaurant-pos`
   - إن وُجد dropdown للفروع: اختر فرعًا نشطًا.
   - `Loading` طبيعي: انتظر skeleton/تحميل.
2. اختر طاولة من مخطط الطاولات
   - المتوقع: لوحة “Current Order” تعرض اسم الطاولة ورسالة إعداد طلب واضحة.
3. من `القائمة` أضف صنفًا:
   - بدون modifiers: يتضاف مباشرة.
   - مع modifiers: افتح modal للتحديد المطلوب ثم Confirm.
4. اضغط `إرسال للمطبخ`
   - المتوقع: Success عربية قصيرة.
   - المتوقع: لا يلزم إعادة اختيار الطاولة للدفع مباشرة من نفس الشاشة.
5. افتح `/kds`
   - المتوقع: تظهر تذكرة جديدة مرتبطة بالفرع والمحطة المناسبة.
6. من بطاقة التذكرة اضغط:
   - `بدء التحضير` ثم `جاهز` ثم `أغلق` (حسب حالة التذكرة).
   - المتوقع: تظهر رسالة Success عربية قصيرة بعد كل تحديث حالة.
7. ارجع إلى `/restaurant-pos`
   - اضغط `الدفع`
   - المتوقع: Success عربية + إغلاق Payment modal + تحديث الحالة + (إن أمكن) طباعة إيصال.

## سيناريو صيدلية (Prescription/OTC)
1. افتح `/pharmacy-pos`
   - اختر الفرع من الـdropdown.
   - راقب banner “التخصيص النهائي للدفعات يتم داخل RPC…”.
2. (Prescription path)
   - اختر تبويب `Prescription`.
   - من “طابور الوصفات” اختر وصفة.
   - من بنود الوصفة اضغط `إضافة للصرف`.
   - اختر batch مناسب من modal (تأكد من expiry badge).
   - من السلة اضغط `صرف الوصفة`.
   - المتوقع: Success عربية + تنظيف السلة + تحديث البيانات.
3. (OTC path)
   - بدّل إلى `OTC`.
   - نفّذ بحث: اكتب اسم/باركود.
   - اضغط `إضافة إلى الصرف` من batch card.
   - كرر اختيار batches متعددة (اختياري).
   - من السلة اضغط `إتمام بيع OTC`.
   - المتوقع: Success عربية + تنظيف السلة + منع خلط prescription/OTC عبر mode switch.

## سيناريو WorkOS (عرض وتجربة سريعة)
1. افتح `/work` (Home)
   - المتوقع: skeleton ثم أرقام summary ورسالة empty إن كانت البيئة بلا بيانات.
2. افتح `/work/search`
   - اكتب query (حرفين على الأقل)
   - المتوقع: نتائج grouped + badges حسب النوع + empty state واضحة عند عدم وجود نتائج.
3. افتح `/work/inbox`
   - اضغط `تعليم الكل كمقروء` إن كان هناك غير مقروءة.
   - المتوقع: Counts تتغير + العناصر تنتقل/تتصفّح بشكل أوضح.
4. افتح `/work/projects/:id` أو `/work/docs`
   - المتوقع: Success عربية بعد Create/Edit/Archive + refresh للعرض.

## أين قد يظهر Empty state “طبيعي”
- المطاعم:
  - `/kds` قبل أي إرسال للمطبخ: “المطبخ جاهز” (طبيعي).
  - `/restaurant-pos` إذا لا توجد طاولات/أصناف لهذا الفرع: رسالة واضحة لتغيير الفرع/إضافة بيانات.
- الصيدلية:
  - `/pharmacy-pos` إذا لم يتم تحميل batches للفرع: رسالة اختيار فرع/لا توجد نتائج.
  - `/patient-med-history` إذا لم يوجد صرف مرتبط بالمريض: رسالة توضح سبب الفراغ.
- WorkOS:
  - `/work` و `/work/search` إذا seed بسيط جدًا أو البيئة بلا بيانات: رسائل توضح أنها ليست Runtime عطلًا.
